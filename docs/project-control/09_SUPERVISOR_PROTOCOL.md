# پروتکل Supervisor

## هدف

Supervisor باید بین درخواست، وضعیت واقعی repository و اجرای Codex سازگاری ایجاد کند. هیچ فازی فقط بر اساس گزارش قبلی «کامل» فرض نمی‌شود.

## ترتیب منابع حقیقت

1. کد و فایل‌های موجود در workspace
2. خروجی تازه `npm test` و `npm run build`
3. route manifest، storage key constants و testها
4. اسناد `docs/project-control`
5. گزارش‌های گفتگو و ادعاهای فازهای قبلی

در تعارض، منبع بالاتر مقدم است و تعارض باید ثبت شود.

## پروتکل شروع P

1. هدف و محدوده ممنوع خوانده شود.
2. `00_MASTER_PLAN`, `01_CURRENT_STATE`, `05_CROSS_IMPACT`, `08_PARKING_LOT` مرور شود.
3. فایل‌های مرتبط از کد کشف شوند.
4. dirty work یا تغییرات ناشناخته حفظ شوند.
5. اثر route، model، storage، backup، drift و test مشخص شود.
6. برنامه کوتاه اجرا ثبت شود.

## پروتکل حین اجرا

- تغییرات خارج از محدوده ممنوع است.
- داده یا فایل کاربر revert نمی‌شود.
- تصمیم مبهم اما پرریسک باید قبل از اجرا روشن شود.
- برای تغییرات destructive، confirmation و recovery لازم است.
- هر تصمیم معماری پایدار در `02_DECISIONS.md` ثبت می‌شود.
- اگر شواهد کافی نیست، عبارت دقیق «نیاز به تأیید دستی» استفاده شود.

## پروتکل پایان P

1. test و build تازه اجرا شود.
2. در frontend، direct route و console حداقل به‌صورت smoke بررسی شود.
3. warningها از failure جدا گزارش شوند.
4. فایل‌ها، routeها، modelها و storage impact فهرست شوند.
5. `01_CURRENT_STATE` و `03_PHASE_LOG` به‌روزرسانی شوند.
6. ریسک و parking lot ثبت شود.
7. پیشنهاد فاز بعدی از ریسک واقعی استخراج شود، نه صرفاً feature بعدی.

## Gate تأیید فاز

Supervisor فقط وقتی فاز را complete اعلام کند که:

- خروجی مورد انتظار واقعاً وجود دارد.
- ممنوعه‌های فاز نقض نشده‌اند.
- TypeScript/build موفق است.
- test موفق است یا دلیل دقیق عدم اجرا ثبت شده است.
- migration یا storage change پنهان وجود ندارد.
- گزارش پایان با کد هم‌خوان است.

## مدیریت اختلاف گزارش و کد

اگر گزارش قبلی با repository ناسازگار بود:

1. repository مبنا قرار گیرد.
2. اختلاف در `01_CURRENT_STATE` ثبت شود.
3. فاز موردنظر «نیازمند تأیید» یا «ناقص» علامت بخورد.
4. از بازسازی حدسی feature بدون درخواست صریح خودداری شود.

## قرارداد تحویل به Codex

هر درخواست بهتر است شامل این موارد باشد:

- فاز و هدف
- وضعیت مبنای تأییدشده
- فایل‌ها یا routeهای مالک
- محدوده مجاز و ممنوع
- acceptance criteria
- test/build مورد انتظار
- قالب گزارش پایان

Codex باید از `06_CODEX_HANDOFF_TEMPLATE.md` برای تحویل‌های طولانی استفاده کند.
