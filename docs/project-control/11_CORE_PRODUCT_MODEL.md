# Core Product Model

آخرین به‌روزرسانی: 2026-06-28

## هدف

`Core Product Model` مدل مفهومی مرکزی کالا در سیستم‌عامل مستر جم است. این سند فقط طراحی کنترل پروژه است و به معنی ساخت UI، migration، route، localStorage key، adapter اجرایی یا ادغام پروژه `mahak-web-version` نیست.

این مدل باید قبل از شروع ماژول کالا، تولید، انبار، اتصال محک و خروجی‌های AI-ready طراحی و تثبیت شود تا همه شاخه‌ها درباره مفهوم «کالا» یک زبان مشترک داشته باشند.

## Core Product Model چیست؟

مدل مرکزی کالا یک قرارداد دامنه‌ای است که مشخص می‌کند هر کالا در مستر جم چگونه شناسایی، دسته‌بندی، قیمت‌گذاری، وزن‌دهی، موجودی‌گیری، تولید، خروجی محک و تحلیل هوشمند می‌شود.

این مدل باید مستقل از UI و مستقل از پروژه کالا باشد. پروژه `mahak-web-version` فقط یک منبع الهام و استخراج آینده است، نه منبعی برای merge مستقیم.

## فیلدهای اصلی پیشنهادی کالا

| فیلد | هدف | توضیح |
|---|---|---|
| `id` | شناسه داخلی | شناسه پایدار داخل مستر جم |
| `productCode` | کد کالا | کد قابل نمایش/جست‌وجو برای عملیات داخلی |
| `mahakCode` | کد محک | شناسه یا کد متناظر برای خروجی/ورودی محک، در صورت وجود |
| `barcode` | بارکد اصلی | بارکد قابل چاپ/اسکن؛ می‌تواند چندتایی شود |
| `name` | نام کالا | عنوان اصلی کالا |
| `displayName` | نام نمایشی | نام انسانی‌تر برای UI یا خروجی |
| `groupCode` | کد گروه | اتصال به گروه کالا و ساختار محک |
| `groupName` | نام گروه | گروه تجاری/عملیاتی کالا |
| `stoneType` | نوع سنگ | اتصال به بانک سنگ |
| `stoneAttributes` | ویژگی‌های سنگ | رنگ، کیفیت، ابعاد، تراش یا ویژگی‌های قابل تحلیل |
| `weight` | وزن | وزن پایه کالا یا سنگ |
| `weightUnit` | واحد وزن | گرم، قیراط یا واحد مصوب آینده |
| `laborCost` | اجرت | اجرت ساخت/پرداخت/تولید |
| `basePrice` | قیمت پایه | قیمت بدون محاسبات لحظه‌ای یا مالیات |
| `salePrice` | قیمت فروش | قیمت قابل استفاده در فروش یا خروجی |
| `currency` | واحد پول | پیش‌فرض آینده باید با مدل مالی هماهنگ شود |
| `inventoryStatus` | وضعیت موجودی | موجود، رزرو، تمام‌شده، در تولید، نیازمند بررسی |
| `defaultInventoryLocation` | محل پیش‌فرض موجودی | انبار، فروشگاه، تولید یا محل عملیاتی |
| `productionStatus` | وضعیت تولید | آماده، در تولید، نیازمند تکمیل، ضایعات/نامعتبر |
| `isActive` | وضعیت فعال | کالاهای غیرفعال حذف فیزیکی نشوند |
| `createdAt` / `updatedAt` | رد تغییرات | تاریخ ایجاد و آخرین تغییر |
| `source` | منبع داده | manual، mahak، import، AI-ready export یا adapter آینده |
| `notes` | توضیحات | توضیح انسانی کوتاه |

## ارتباط کالا با دامنه‌های دیگر

| ارتباط | نقش در مدل مرکزی |
|---|---|
| بارکد | باید به کالا وصل باشد، اما امکان چند بارکد برای یک کالا یا alias آینده محفوظ بماند. |
| گروه کالا | گروه کالا باید جدا از نام کالا باشد تا خروجی محک، گزارش فروش و تحلیل موجودی پایدار بماند. |
| سنگ | بانک سنگ باید منبع ویژگی‌های سنگ باشد؛ کالا فقط به نوع/ویژگی یا شناسه سنگ ارجاع دهد. |
| وزن | وزن باید عدد و واحد داشته باشد و پایه محاسبات قیمت، تولید و انبار شود. |
| اجرت | اجرت نباید با قیمت پایه یکی شود؛ برای تولید و فروش نیازمند فیلد مستقل است. |
| قیمت | قیمت پایه، قیمت فروش و currency باید قابل تفکیک باشند تا بعدها Finance به آن وصل شود. |
| موجودی | کالا فقط موجودیت محصول است؛ مقدار موجودی باید در `Core Inventory Model` یا eventهای انبار نگه‌داری شود. |
| تولید | تولید باید از کالا استفاده کند، اما جریان تولید در `Production Flow Model` جدا طراحی شود. |
| خروجی محک | مدل باید کد گروه، کد کالا، نام، بارکد، قیمت و وزن را طوری نگه دارد که export محک قابل mapping باشد. |
| AI-ready export | خروجی تحلیلی باید داده تمیز، بدون duplicate، با شناسه پایدار، گروه، ویژگی سنگ، وزن و قیمت داشته باشد. |

## چیزهایی که از پروژه کالا بعداً باید استخراج شود

| موضوع | نوع استخراج | توضیح |
|---|---|---|
| مدل کالا | Extract model | بررسی ساختار موجود کالا و تطبیق با مدل مرکزی |
| بارکد | Extract model / Extract idea | بررسی تولید، نگه‌داری و کنترل بارکد |
| بانک سنگ | Extract model / Extract schema | استخراج مفهوم نوع سنگ و ویژگی‌های اصلی |
| کد گروه | Extract schema | بررسی mapping گروه‌ها برای محک و گزارش |
| خروجی Excel محک | Extract adapter idea | بررسی ستون‌ها، ترتیب، قالب و قوانین خروجی |
| کنترل تکراری | Extract logic idea | بررسی روش تشخیص کالا/بارکد/کد تکراری |
| خروجی قابل تحلیل برای AI | Extract schema / Extract idea | طراحی export تمیز برای تحلیل و آموزش/پرس‌وجوی آینده |

## چیزهایی که فعلاً ممنوع است

- merge مستقیم `mahak-web-version`
- تغییر database اصلی
- ساخت ماژول کالا در UI اصلی
- ساخت migration
- تغییر route
- تغییر localStorage
- ساخت adapter اجرایی محک
- import داده واقعی کالا به پروژه اصلی
- تغییر auth، backend یا sync به بهانه کالا

## ترتیب امن آینده

1. مدل مفهومی کالا ثبت و بازبینی شود.
2. `Product Schema Draft` طراحی شود.
3. `Product Adapter Boundary` طراحی شود.
4. خروجی‌ها و mappingها روی داده نمونه و غیرواقعی تست شوند.
5. UI اولیه فقط بعد از تأیید مدل و schema طراحی شود.
6. migration فقط در فاز مستقل و با اجازه مرکز فرمان انجام شود.

## پرسش‌های باز برای مرکز کنترل

- واحد وزن اصلی مستر جم چیست و آیا چند واحد باید همزمان پشتیبانی شود؟
- کد کالا داخلی با کد محک یکی است یا دو شناسه جدا باید بمانند؟
- بارکد چندتایی برای یک کالا لازم است یا فقط یک بارکد اصلی کافی است؟
- قیمت فروش از مدل کالا می‌آید یا از `Core Financial Event Model` و قیمت‌گذاری آینده؟
- بانک سنگ باید مستقل باشد یا زیرمجموعه Product باقی بماند؟

## پیشنهاد مرکز کنترل

فعلاً فقط طراحی ادامه پیدا کند. قدم بعدی امن، ساخت `Product Schema Draft` و `Product Adapter Boundary` در docs است، بدون کدنویسی، بدون migration و بدون merge پروژه کالا.
