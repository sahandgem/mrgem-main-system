# Core Financial Event Model

آخرین به‌روزرسانی: 2026-06-28

## هدف

`Core Financial Event Model` مدل مفهومی مرکزی برای ثبت، تحلیل و کنترل رویدادهای مالی مستر جم است. این سند فقط طراحی کنترل پروژه است و به معنی ساخت UI مالی، تغییر auth، migration، route، localStorage key، database یا ادغام پروژه `audit-app` نیست.

این مدل باید قبل از اتصال پروژه پول، رسیدهای عکس‌دار، اپ موبایل، داشبورد نقدینگی، حقوق، فروش، خرید، تولید و اتصال بانکی طراحی و تأیید شود.

## Core Financial Event Model چیست؟

مدل مرکزی رویداد مالی یک قرارداد دامنه‌ای است که هر رخداد مالی را به شکل قابل ردیابی، قابل تأیید، قابل اتصال به سند/رسید و قابل تحلیل ثبت می‌کند.

در این مدل، «پول» فقط یک عدد نیست؛ هر تغییر مالی باید منشأ، مقصد، وضعیت پرداخت، وضعیت تأیید، طرف حساب، سند، ارتباط عملیاتی و منبع داده داشته باشد.

## رویداد مالی در مستر جم یعنی چه؟

رویداد مالی یعنی هر اتفاقی که وضعیت پول، بدهی، طلب، نقدینگی، هزینه، درآمد، حقوق، خرید، فروش یا تولید را تغییر دهد یا نیازمند تأیید مالی/مدیریتی باشد.

رویداد مالی می‌تواند از UI آینده، ثبت رسید، اپ موبایل، import بانکی، فروش کالا، خرید کالا، حقوق کارکنان، هزینه تولید یا adapter آینده پروژه پول وارد شود.

## انواع رویداد مالی

| نوع رویداد | توضیح |
|---|---|
| دریافت | ورود پول از مشتری، طرف حساب، بانک یا منبع دیگر |
| پرداخت | خروج پول به شخص، تأمین‌کننده، کارمند یا حساب دیگر |
| خرید | ثبت تعهد یا پرداخت بابت خرید کالا/مواد/خدمات |
| فروش | ثبت درآمد یا طلب ناشی از فروش کالا/خدمت |
| هزینه | هزینه عمومی، عملیاتی، اداری یا متفرقه |
| حقوق | پرداخت یا تعهد حقوق/مزایا به کارکنان |
| بدهی | تعهد پرداخت به شخص یا سازمان |
| طلب | مبلغ قابل دریافت از شخص یا سازمان |
| انتقال بانکی | جابه‌جایی پول بین حساب‌ها یا صندوق‌ها |
| چک | رویداد چک دریافتی/پرداختی و وضعیت آن |
| نقدینگی | رویداد مؤثر بر وضعیت نقدینگی یا بحران نقدینگی |
| هزینه تولید | هزینه مستقیم یا غیرمستقیم وابسته به تولید |
| اصلاحیه مالی | اصلاح، تعدیل یا برگشت رویداد قبلی |
| تایید مدیر | رویداد یا وضعیت تأیید/رد/نیاز به بررسی مدیر |

## فیلدهای اصلی پیشنهادی

| فیلد | هدف | توضیح |
|---|---|---|
| `id` | شناسه داخلی | شناسه پایدار رویداد مالی |
| `eventType` | نوع رویداد | دریافت، پرداخت، خرید، فروش، هزینه، حقوق، بدهی، طلب و غیره |
| `amount` | مبلغ | مقدار عددی رویداد |
| `currency` | ارز | واحد پول، پیش‌فرض آینده باید مصوب شود |
| `eventDate` | تاریخ رویداد | تاریخ وقوع مالی |
| `counterpartyId` | شخص/طرف حساب | مشتری، فروشنده، کارمند، شریک یا طرف بیرونی |
| `counterpartyName` | نام طرف حساب | نام نمایشی برای audit و گزارش |
| `moneySource` | منبع پول | حساب بانکی، صندوق، کیف پول، شخص، یا منبع عملیاتی |
| `moneyDestination` | مقصد پول | حساب، صندوق، شخص، تولید، خرید یا هزینه |
| `paymentStatus` | وضعیت پرداخت | پرداخت‌شده، پرداخت‌نشده، جزئی، سررسیددار، برگشتی |
| `approvalStatus` | وضعیت تأیید | پیش‌نویس، در انتظار تأیید، تأییدشده، ردشده، نیازمند بررسی |
| `description` | توضیحات | شرح انسانی کوتاه |
| `receiptId` | سند یا رسید | اتصال به رسید عکس‌دار یا سند مالی |
| `productId` | ارتباط با کالا | اتصال اختیاری به `Core Product Model` |
| `productionId` | ارتباط با تولید | اتصال اختیاری به جریان تولید |
| `employeeId` | ارتباط با کارمند | حقوق، تنخواه، هزینه یا مسئول ثبت |
| `bankAccountId` | ارتباط با بانک | حساب بانکی یا گردش حساب مرتبط |
| `source` | منبع داده | manual، audit-app، mobile، receipt، bank-import، adapter |
| `createdAt` / `updatedAt` | رد تغییرات | تاریخ ایجاد و آخرین تغییر |
| `createdBy` | ثبت‌کننده | کاربر/نقش ثبت‌کننده در مدل آینده |
| `approvedBy` | تأییدکننده | مدیر یا نقش تأییدکننده در مدل آینده |
| `relatedEventId` | ارتباط اصلاحیه | اتصال اصلاحیه یا برگشت به رویداد اصلی |

## ارتباط با بخش‌های دیگر

| بخش | نقش ارتباط |
|---|---|
| پروژه پول / `audit-app` | منبع الهام برای نقدینگی، schema مالی، roles، RLS، تایید مدیر و داشبورد بحران نقدینگی؛ بدون merge مستقیم |
| نقدینگی | رویدادهای مالی باید بتوانند وضعیت cash-in/cash-out و فشار نقدینگی را تغذیه کنند |
| داشبورد بحران نقدینگی | بعداً باید از رویدادهای مالی و وضعیت پرداخت/سررسید ساخته شود |
| نقش‌ها و تایید مدیر | وضعیت تأیید، ثبت‌کننده، تأییدکننده و جریان review باید در مدل دیده شود؛ auth فعلاً تغییر نمی‌کند |
| رسیدهای عکس‌دار | هر رویداد می‌تواند به رسید یا سند عکس‌دار وصل شود |
| اپ موبایل | موبایل آینده می‌تواند رسید و رویداد اولیه را ثبت کند، اما بعد از مدل رسید و مالی |
| بانک و گردش حساب | mapping تراکنش بانکی باید به رویداد مالی وصل شود، نه اینکه database را مستقیم تغییر دهد |
| خرید کالا | خرید باید به کالا و طرف حساب وصل شود و روی موجودی/تعهد مالی اثر بگذارد |
| فروش کالا | فروش باید به کالا، مشتری، دریافت/طلب و نقدینگی وصل شود |
| حقوق کارکنان | پرداخت حقوق باید با کارمند، وضعیت پرداخت و تأیید مدیر مرتبط باشد |
| تولید | هزینه تولید باید به batch/flow تولید و کالاهای مرتبط وصل شود |

## چیزهایی که از پروژه پول بعداً باید استخراج شود

| موضوع | نوع استخراج | توضیح |
|---|---|---|
| مدل نقدینگی | Extract model | بررسی مفهوم cash pressure، ورودی/خروجی و وضعیت بحران |
| schema مالی | Extract schema | بررسی ساختار رویدادها، تراکنش‌ها، تأییدها و گزارش‌ها |
| نقش‌ها | Extract idea / Extract schema | بررسی نقش‌های مدیر، ثبت‌کننده، مشاهده‌گر و تأییدکننده |
| RLS | Extract policy idea | بررسی سیاست‌های دسترسی، بدون تغییر auth اصلی |
| تایید مدیر | Extract workflow idea | بررسی جریان ثبت، review، approve/reject |
| داشبورد بحران نقدینگی | Extract UI pattern / Extract logic idea | بررسی KPIها، alertها و اولویت‌بندی بحران |
| جریان ثبت/بررسی/تایید | Extract workflow idea | طراحی flow آینده برای رویداد مالی و رسید |

## ترتیب امن آینده

1. مدل مفهومی رویداد مالی ثبت و بازبینی شود.
2. `Financial Schema Draft` طراحی شود.
3. `Financial Adapter Boundary` طراحی شود.
4. مدل با داده نمونه و غیرواقعی تست شود.
5. UI مالی اولیه فقط بعد از تأیید schema طراحی شود.
6. اتصال رسیدها و receipt attachment flow طراحی شود.
7. migration فقط در فاز مستقل و با اجازه مرکز فرمان انجام شود.

## پرسش‌های باز برای مرکز کنترل

- ارز پایه مستر جم چیست و آیا چندارزی لازم است؟
- طرف حساب‌ها در مدل مستقل هستند یا ابتدا به صورت نام/شناسه سبک ثبت می‌شوند؟
- وضعیت تأیید مدیر چندمرحله‌ای است یا یک سطح تأیید کافی است؟
- رسید عکس‌دار باید قبل از رویداد مالی ثبت شود یا بعد از آن هم قابل پیوست باشد؟
- mapping تراکنش بانکی دستی است یا adapter آینده خواهد داشت؟
- رویداد حقوق کارکنان در Finance می‌ماند یا به ماژول HR/Payroll آینده وصل می‌شود؟
