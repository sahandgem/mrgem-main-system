# Multi Codex Handoff Protocol

آخرین به‌روزرسانی: 2026-06-29

## هدف

این سند پروتکل تحویل کار بین چند Codex، چند repo و مرکز فرمان مستر جم را ثبت می‌کند. هدف جلوگیری از کار موازی مخرب، تغییر ناخواسته فایل‌های مشترک و merge بدون کنترل است.

## قانون شروع کار

هر Codex قبل از شروع باید:

1. `git status` بگیرد.
2. اگر working tree تمیز نبود، هیچ کاری نکند و گزارش بدهد.
3. محدوده مجاز فایل‌ها و ممنوعیت‌ها را از دستور همان فاز بخواند.
4. فقط در همان محدوده کار کند.
5. اگر برای انجام کار نیاز به تغییر خارج از محدوده پیدا کرد، متوقف شود و گزارش بدهد.

## محدوده کار هر Codex

- Codex باید فقط فایل‌های مربوط به task خودش را تغییر دهد.
- docs، code، route، model، storage و migration باید جدا و با فازهای مشخص تغییر کنند.
- تغییرات cross-module بدون تأیید مرکز فرمان ممنوع است.
- خروجی labها فقط به صورت گزارش، schema، model یا pattern به main پیشنهاد می‌شود.

## قالب گزارش استاندارد

هر خروجی Codex باید شامل این موارد باشد:

- فایل‌های تغییرکرده
- کد اجرایی تغییر کرد یا نه
- docs تغییر کرد یا نه
- test/build اگر لازم بود
- commit hash
- push موفق بود یا نه
- هشدار معماری
- پیشنهاد قدم بعد

## ممنوعیت‌های بدون اجازه مرکز فرمان

هیچ Codex حق ندارد بدون اجازه مرکز فرمان:

- auth تغییر دهد
- database تغییر دهد
- migration بسازد
- route جدید بسازد
- localStorage key جدید بسازد
- پروژه پول را merge کند
- پروژه کالا را merge کند
- زیرپروژه‌ها را مستقیم وارد main کند
- فایل مشترک در حال کار Codex دیگر را تغییر دهد

## Handoff بین Codexها

هر handoff باید شامل این اطلاعات باشد:

| بخش | توضیح |
|---|---|
| هدف کار | فاز یا خروجی مورد انتظار |
| محدوده مجاز | فایل‌ها، پوشه‌ها یا repoهای قابل تغییر |
| محدوده ممنوع | route، storage، db، migration، UI یا کد حساس |
| وضعیت Git | clean/dirty و آخرین commit |
| فایل‌های تغییرکرده | لیست دقیق |
| تست و build | نتیجه یا دلیل عدم اجرا |
| ریسک باقی‌مانده | مواردی که Codex بعدی باید بداند |
| پیشنهاد بعدی | قدم کنترل‌شده بعد |

## قانون conflict

اگر Codex هنگام شروع ببیند working tree dirty است:

- هیچ تغییری انجام ندهد.
- فایل‌های dirty را گزارش کند.
- حدس نزند تغییرات متعلق به کیست.
- commit، reset، checkout یا merge انجام ندهد مگر دستور صریح مرکز فرمان داشته باشد.

## قانون commit

- commit باید کوچک و قابل فهم باشد.
- message باید scope و هدف را نشان دهد.
- docs و کد اجرایی تا حد ممکن در commitهای جدا باشند.
- push فقط پس از اطمینان از محدوده تغییرات مجاز انجام شود.

## پیشنهاد مرکز کنترل

قبل از هر فاز جدید، دستور باید دقیقاً مشخص کند: repo، branch، فایل‌های قابل تغییر، محدوده ممنوع، test/build لازم و commit message.
