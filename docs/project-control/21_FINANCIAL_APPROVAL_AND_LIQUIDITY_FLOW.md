# Financial Approval And Liquidity Flow

آخرین به‌روزرسانی: 2026-06-29

## هدف

این سند جریان ثبت، بررسی، تأیید مدیر و مدل نقدینگی آینده را طراحی می‌کند. این کار فقط مستند است و هیچ کد اجرایی، auth، RLS، database، migration، UI یا merge مستقیم `audit-app` ایجاد نمی‌کند.

## جریان ثبت مالی

1. رویداد مالی به صورت draft ساخته می‌شود.
2. منبع داده مشخص می‌شود: manual، receipt، mobile، bank-import، audit-app یا adapter.
3. فیلدهای اصلی validate می‌شوند.
4. اگر سند یا رسید وجود دارد، به رویداد وصل می‌شود.
5. اگر تراکنش بانکی وجود دارد، mapping پیشنهادی ساخته می‌شود.
6. رویداد وارد وضعیت `pending_review` یا `pending_manager` می‌شود.

## جریان بررسی مالی

بررسی‌کننده آینده باید بتواند موارد زیر را کنترل کند:

- مبلغ معتبر است یا نه
- تاریخ با سند/بانک سازگار است یا نه
- طرف حساب مشخص است یا نه
- paymentStatus درست است یا نه
- receipt یا bank transaction mismatch دارد یا نه
- رویداد duplicate نیست
- نیاز به تأیید مدیر دارد یا نه

## جریان تایید مدیر

مدیر آینده می‌تواند یکی از تصمیم‌های زیر را ثبت کند:

- approve
- reject
- request revision
- mark as paid
- hold
- escalate

تأیید مدیر نباید auth یا RLS واقعی را در این فاز تغییر دهد؛ فقط مدل مفهومی تصمیم ثبت می‌شود.

## مدل نقدینگی

Liquidity model باید از رویدادهای مالی، وضعیت پرداخت، تاریخ سررسید، دریافت‌ها، پرداخت‌ها و تعهدات آینده سیگنال بسازد.

سیگنال‌های پیشنهادی:

- cash in
- cash out
- overdue payment
- upcoming due
- unpaid expense
- expected receipt
- blocked cash
- liquidity pressure

## هشدار بحران نقدینگی

هشدار نقدینگی باید فقط بعد از طراحی schema و adapter ساخته شود. معیارهای مفهومی:

- پرداخت‌های نزدیک بدون منبع نقدی کافی
- دریافت‌های عقب‌افتاده
- هزینه‌های تأییدنشده اما ضروری
- چک‌های نزدیک به سررسید
- اختلاف شدید cash-in و cash-out
- افزایش تعهدات پرداخت نسبت به دریافت‌های قطعی

## نقش‌های احتمالی

| نقش | مسئولیت |
|---|---|
| recorder | ثبت اولیه رویداد، رسید یا داده بانکی |
| reviewer | بررسی صحت داده، سند، تطبیق و duplicate |
| manager | تأیید، رد، hold یا درخواست اصلاح |
| admin | مدیریت سیاست‌ها، دسترسی‌ها و تنظیمات آینده |

## چیزهایی که بعداً از audit-app بررسی می‌شود

| موضوع | نوع بررسی |
|---|---|
| roles | استخراج ایده نقش‌ها و سطح دسترسی |
| RLS | بررسی policyها بدون تغییر auth اصلی |
| approval | بررسی flow تأیید مدیر |
| liquidity dashboard | بررسی KPIها، هشدارها و اولویت‌بندی بحران |

## قواعد توقف

- اگر approval نیاز به auth واقعی پیدا کند، فاز باید متوقف شود.
- اگر liquidity نیاز به database/migration پیدا کند، فاز باید جدا تعریف شود.
- هیچ RLS واقعی بدون تصمیم مرکز فرمان ساخته نشود.
- dashboard نقدینگی قبل از schema و adapter نباید وارد main شود.

## پیشنهاد مرکز کنترل

قدم بعدی امن، طراحی `Financial Validation Rules` و `Financial Import Decision Flow` است. اتصال auth/RLS و dashboard نقدینگی باید بعداً و با فاز جدا انجام شود.
