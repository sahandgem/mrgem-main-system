# Receipt And Bank Transaction Mapping

آخرین به‌روزرسانی: 2026-06-29

## هدف

این سند رابطه رسید عکس‌دار، رویداد مالی، گردش بانکی و اپ موبایل آینده را طراحی می‌کند. این کار فقط مستند است و هیچ کد اجرایی، فایل ذخیره‌سازی، UI، auth، database، migration یا اتصال بانکی ایجاد نمی‌کند.

## رسید عکس‌دار چیست؟

رسید عکس‌دار سند تصویری یا فایل پیوست برای یک رخداد مالی است. رسید ممکن است فاکتور، عکس کارت‌خوان، رسید انتقال، قبض، چک، سند خرید یا هزینه باشد.

رسید به تنهایی رویداد مالی تأییدشده نیست؛ باید به یک `FinancialEvent` وصل شود و توسط reviewer/manager آینده بررسی شود.

## اتصال رسید به رویداد مالی

هر رسید آینده باید حداقل این داده‌ها را داشته باشد:

- `receiptId`
- فایل یا مرجع فایل
- تاریخ ثبت
- مبلغ خوانده‌شده یا واردشده
- توضیح
- ثبت‌کننده
- وضعیت بررسی
- رویداد مالی مرتبط

قواعد اتصال:

- یک receipt می‌تواند به یک رویداد مالی وصل شود.
- یک رویداد مالی ممکن است چند receipt داشته باشد.
- اختلاف مبلغ receipt و event باید warning بسازد.
- receipt بدون event باید در queue بررسی باقی بماند.

## mapping گردش بانکی

گردش بانکی باید به event مالی map شود، نه اینکه مستقیماً داده مالی را تغییر دهد.

فیلدهای mapping:

- شماره پیگیری
- تاریخ
- مبلغ
- مبدا
- مقصد
- توضیحات بانک
- نوع احتمالی: پرداخت یا دریافت
- event مالی پیشنهادی
- سطح اطمینان mapping

## تشخیص پرداخت/دریافت

تشخیص مفهومی:

- مبلغ ورودی به حساب یعنی دریافت احتمالی.
- مبلغ خروجی از حساب یعنی پرداخت احتمالی.
- توضیحات بانک، مبدا، مقصد و counterparty برای تشخیص دقیق‌تر لازم است.
- تراکنش‌های داخلی بین حساب‌ها باید `bank_transfer` شوند، نه درآمد یا هزینه.

## شماره پیگیری، تاریخ، مبلغ، مبدا، مقصد، توضیحات

این فیلدها برای جلوگیری از duplicate و خطای انسانی ضروری هستند:

| فیلد | نقش |
|---|---|
| شماره پیگیری | تشخیص تراکنش تکراری یا mapping قطعی |
| تاریخ | تطبیق با event و receipt |
| مبلغ | کنترل mismatch |
| مبدا | تشخیص پرداخت‌کننده یا حساب خروجی |
| مقصد | تشخیص دریافت‌کننده یا حساب ورودی |
| توضیحات | کمک به تشخیص طرف حساب و نوع رویداد |

## ارتباط با اپ موبایل آینده

اپ موبایل آینده می‌تواند receipt capture انجام دهد، اما فقط پس از تثبیت مدل مالی و receipt flow.

سناریوی آینده:

1. کارمند عکس رسید را ثبت می‌کند.
2. اپ موبایل metadata اولیه می‌گیرد.
3. رسید وارد queue بررسی می‌شود.
4. reviewer آن را به event مالی وصل می‌کند.
5. manager در صورت نیاز تأیید می‌کند.

## خطرهای خطای انسانی

| خطر | کنترل پیشنهادی |
|---|---|
| مبلغ اشتباه وارد شود | تطبیق با receipt/bank transaction |
| تاریخ اشتباه باشد | warning اختلاف تاریخ |
| receipt به event اشتباه وصل شود | review و undo/ correction log آینده |
| پرداخت و دریافت اشتباه تشخیص داده شود | mapping با confidence و review |
| تراکنش بانکی duplicate شود | شماره پیگیری و مبلغ/تاریخ |
| عکس رسید ناخوانا باشد | status نیازمند عکس بهتر یا بررسی دستی |

## قواعد توقف

- receipt بدون event نباید وارد گزارش مالی قطعی شود.
- bank transaction با confidence پایین نباید خودکار map شود.
- mobile capture نباید قبل از مدل مالی و receipt flow ساخته شود.
- import بانکی واقعی بدون adapter و approval ممنوع است.

## پیشنهاد مرکز کنترل

قدم بعدی امن، طراحی `Receipt Validation Rules` و `Bank Transaction Matching Rules` است. اپ موبایل و import بانکی واقعی باید بعداً و با فاز جدا طراحی شوند.
