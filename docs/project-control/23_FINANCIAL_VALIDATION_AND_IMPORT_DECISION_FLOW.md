# Financial Validation and Import Decision Flow

آخرین به‌روزرسانی: 2026-06-29

## هدف

این سند مشخص می‌کند داده مالی قبل از ورود به مستر جم چگونه بررسی شود و چه زمانی اجازه ثبت، اتصال، اصلاح یا توقف داشته باشد. این طراحی فقط مستند است و هیچ کد اجرایی، database، migration، route، UI یا localStorage ایجاد نمی‌کند.

## Financial Validation Rules

هر رویداد مالی آینده باید قبل از ثبت نهایی از این کنترل‌ها عبور کند:

| حوزه | قانون اعتبارسنجی | سطح پیش‌فرض |
|---|---|---|
| نوع رویداد | نوع باید از دسته‌های مصوب مثل دریافت، پرداخت، خرید، فروش، هزینه، حقوق، بدهی، طلب، انتقال بانکی، چک، نقدینگی، هزینه تولید، اصلاحیه یا تایید مدیر باشد. | Error |
| مبلغ | مبلغ باید عددی، مثبت و با واحد پول مشخص باشد. | Error |
| تاریخ | تاریخ رویداد باید معتبر و قابل مقایسه با تاریخ بانک یا رسید باشد. | Error |
| طرف حساب | برای رویدادهای دارای شخص، مشتری، فروشنده یا کارمند، طرف حساب باید قابل تشخیص باشد یا به review برود. | Warning |
| منبع و مقصد پول | برداشت و واریز باید منبع/مقصد سازگار داشته باشند. | Warning |
| وضعیت پرداخت | وضعیت پرداخت باید در مقدارهای مجاز مثل draft، pending، paid، partial، failed یا reversed باشد. | Error |
| وضعیت تایید | رویداد حساس بدون تایید مدیر نباید نهایی شود. | Needs manager review |
| رسید | اگر نوع رویداد نیازمند سند است، نبود رسید باید هشدار یا توقف ایجاد کند. | Warning |
| گردش بانکی | اگر bank transaction وجود دارد، مبلغ، تاریخ و شماره پیگیری باید با رویداد سازگار باشد. | Warning |
| تکراری بودن | تراکنش با شماره پیگیری، مبلغ، تاریخ، حساب و توضیح مشابه باید duplicate candidate شود. | Warning |
| منبع داده | dataSource باید مشخص کند داده از ثبت دستی، Excel بانک، receipt، audit-app یا import آینده آمده است. | Error |
| audit trail | هر تصمیم import، اصلاح یا تایید باید قابل ردیابی باشد. | Error |

## سطح خطاها

| سطح | معنی | رفتار پیشنهادی |
|---|---|---|
| Error | داده ناقص یا ناسازگار است و ثبت امن نیست. | import blocked |
| Warning | داده قابل ثبت اولیه است اما ریسک دارد. | needs review |
| Info | نکته غیرمسدودکننده برای آگاهی مدیر یا reviewer. | import allowed با یادداشت |
| Needs manager review | تصمیم حساس است و بدون تایید مدیر نباید نهایی شود. | نگه‌داری در صف بررسی |

## Financial Import Decision Flow

1. دریافت داده از منبع آینده مثل ثبت دستی، Excel بانک، رسید عکس‌دار یا adapter.
2. normalize اولیه فیلدها: مبلغ، تاریخ، نوع تراکنش، توضیحات، مبدا، مقصد و شماره پیگیری.
3. اجرای validation rules.
4. بررسی duplicate و match با رویدادهای موجود.
5. تعیین سطح اطمینان و نیاز به تایید مدیر.
6. ثبت تصمیم import در audit trail.
7. ارسال خروجی به یکی از تصمیم‌های ورود.

## تصمیم‌های ورود

| تصمیم | کاربرد |
|---|---|
| Import allowed | داده کامل و کم‌ریسک است و می‌تواند به رویداد جدید یا موجود تبدیل شود. |
| Import blocked | خطای جدی وجود دارد و ورود داده ممنوع است. |
| Needs review | داده قابل بررسی است اما برای ثبت نهایی به reviewer یا مدیر نیاز دارد. |
| Attach to existing event | تراکنش یا رسید به رویداد مالی موجود وصل می‌شود. |
| Create new event | رویداد مالی جدید ساخته می‌شود. |
| Confirm installment | قسط فقط با match بسیار قوی و قانون مصوب مدیر تایید می‌شود. |
| Correction required | داده باید قبل از ورود اصلاح شود. |
| Duplicate candidate | احتمال تکراری بودن وجود دارد و نباید خودکار ثبت شود. |

## قوانین توقف

- هیچ import واقعی قبل از adapter boundary و validation اجرایی مجاز نیست.
- هیچ auto confirm برای قسط بدون اطمینان خیلی بالا و قانون مصوب مدیر مجاز نیست.
- هیچ تراکنش duplicate candidate نباید خودکار رویداد جدید بسازد.
- هیچ تغییر database، auth، migration، route یا localStorage در این فاز مجاز نیست.
