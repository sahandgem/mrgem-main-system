# Receipt Review and Bank Match Confidence

آخرین به‌روزرسانی: 2026-06-29

## هدف

این سند تعریف می‌کند رسید عکس‌دار و گردش بانکی آینده چگونه به رویداد مالی وصل شوند و سطح اطمینان match چگونه تعیین شود. این مرحله فقط طراحی مستند است و هیچ کد اجرایی یا اتصال واقعی ایجاد نمی‌کند.

## Receipt Review Flow

رسید عکس‌دار سند پشتیبان رویداد مالی است، نه منبع قطعی حقیقت. جریان بررسی پیشنهادی:

1. دریافت رسید از ثبت دستی، اپ موبایل آینده یا import کنترل‌شده.
2. استخراج یا ثبت دستی اطلاعات پایه: مبلغ، تاریخ، شماره پیگیری، طرف حساب، توضیح و نوع احتمالی رویداد.
3. مقایسه با رویداد مالی موجود یا گردش بانکی.
4. تعیین سطح اطمینان match.
5. ارسال به تصمیم: اتصال، ایجاد رویداد، نیاز به review، conflict یا no match.
6. ثبت تصمیم reviewer یا مدیر در audit trail.

## اتصال رسید به رویداد مالی

رسید باید بتواند یکی از این وضعیت‌ها را داشته باشد:

| وضعیت | معنی |
|---|---|
| Unlinked | هنوز به رویداد مالی وصل نشده است. |
| Suggested match | سیستم یک یا چند رویداد احتمالی پیشنهاد کرده است. |
| Attached | رسید به رویداد مالی تاییدشده وصل شده است. |
| Conflict | اطلاعات رسید با رویداد یا بانک تعارض دارد. |
| Rejected | رسید نامعتبر یا مربوط به رویداد دیگری است. |

## Bank Match Confidence

| سطح اطمینان | معیار کلی | تصمیم پیشنهادی |
|---|---|---|
| Exact match | مبلغ، تاریخ، شماره پیگیری، مبدا/مقصد و نوع رویداد همگی همخوان هستند. | attach یا confirm با ثبت audit |
| Strong match | مبلغ و تاریخ دقیق هستند و حداقل یک نشانه قوی مثل شماره پیگیری یا شرح بانکی سازگار است. | attach پیشنهادی یا review سبک |
| Weak match | فقط بخشی از شواهد مثل مبلغ و شباهت توضیح وجود دارد. | needs review |
| Conflict | مبلغ، تاریخ، طرف حساب یا جهت تراکنش تناقض دارد. | blocked یا correction required |
| No match | هیچ رویداد یا تراکنش قابل اتکایی پیدا نشد. | create candidate یا review |

## معیارهای match

| معیار | نقش در match |
|---|---|
| مبلغ | مهم‌ترین معیار؛ اختلاف مبلغ معمولاً conflict ایجاد می‌کند. |
| تاریخ | باید در همان روز یا بازه مجاز تعریف‌شده باشد. |
| شماره پیگیری | نشانه قوی برای exact یا strong match است. |
| مبدا | حساب یا شخص پرداخت‌کننده را کنترل می‌کند. |
| مقصد | حساب یا شخص دریافت‌کننده را کنترل می‌کند. |
| توضیحات | برای تشخیص قسط، خرید، دریافت، هزینه یا طرف حساب استفاده می‌شود. |
| طرف حساب | از اشتباه اتصال تراکنش به فرد یا شرکت غلط جلوگیری می‌کند. |
| نوع رویداد | جهت برداشت/واریز باید با نوع رویداد سازگار باشد. |
| شباهت شرح بانکی | برای matchهای غیرقطعی کاربرد دارد و نباید تنها معیار auto confirm باشد. |

## ریسک‌های اصلی

- اتصال رسید به رویداد اشتباه.
- تشخیص اشتباه تراکنش از روی شرح بانکی ناقص.
- اختلاف تاریخ ثبت رسید با تاریخ بانک.
- ثبت دوباره یک پرداخت به دلیل نبود شماره پیگیری.
- تایید پرداخت بدون سند کافی.

## تصمیم‌های ممنوع فعلی

- اتصال خودکار رسید حساس بدون review.
- تایید قسط فقط بر اساس شباهت متن توضیحات.
- ساخت mobile receipt capture یا bank import اجرایی قبل از مدل و validator.
- تغییر database، auth، route، localStorage یا migration.
