# Financial Liquidity Alert and Approval Rules

آخرین به‌روزرسانی: 2026-06-29

## هدف

این سند مرز هشدار نقدینگی و تایید مدیر را برای آینده مالی مستر جم مشخص می‌کند. هدف جلوگیری از پرداخت بدون کنترل، اختلاف رسید و بانک، اصلاحیه مالی بی‌ردپا و هشدارهای دیرهنگام نقدینگی است.

## Liquidity Alert Rules

هشدار نقدینگی باید از ترکیب cash-in، cash-out، سررسیدها، بدهی‌ها، اقساط، هزینه‌های تولید و مانده‌های بانکی ساخته شود.

| سیگنال | توضیح | سطح پیشنهادی |
|---|---|---|
| کاهش مانده نقد | مانده قابل استفاده کمتر از حد امن آینده شود. | Warning |
| سررسید نزدیک | پرداخت یا قسط نزدیک است و cash-in کافی دیده نمی‌شود. | Warning |
| سررسید عقب‌افتاده | بدهی یا قسط از تاریخ مجاز عبور کرده است. | Error |
| cash-out غیرعادی | برداشت بزرگ یا خارج از الگوی معمول ثبت شود. | Needs manager review |
| receipt/bank mismatch | رسید و بانک از نظر مبلغ، تاریخ یا طرف حساب ناسازگار باشند. | Error |
| پرداخت بدون سند | پرداخت حساس بدون رسید یا bank match ثبت شود. | Warning |
| اصلاحیه مالی | تغییر مبلغ، تاریخ، طرف حساب یا وضعیت تایید بعد از ثبت اولیه رخ دهد. | Needs manager review |

## Manager Approval Boundary

پرداخت‌ها و اصلاح‌های زیر باید در آینده به تایید مدیر نیاز داشته باشند:

- پرداخت بالاتر از سقف مصوب آینده.
- پرداخت بدون رسید یا bank transaction معتبر.
- پرداخت مرتبط با بدهی، حقوق، چک یا هزینه تولید حساس.
- تایید قسط با match کمتر از exact یا strong.
- اصلاحیه مالی که مبلغ، طرف حساب، تاریخ یا منبع پول را تغییر دهد.
- حذف، برگشت یا reverse کردن رویداد مالی.
- اتصال دستی تراکنش بانکی به رویداد حساس.

## اختلاف رسید و بانک

اگر رسید و بانک اختلاف داشته باشند:

1. رویداد نباید auto confirm شود.
2. اختلاف باید به عنوان `Conflict` ثبت شود.
3. reviewer باید علت اختلاف را مشخص کند.
4. اگر اصلاح لازم است، correction request ثبت شود.
5. تصمیم نهایی باید در audit trail باقی بماند.

## پرداخت بدون سند

پرداخت بدون سند فقط می‌تواند به صورت draft یا pending review بماند، مگر اینکه مرکز فرمان در آینده قانون خاصی تعریف کند. برای پرداخت‌های حساس، نبود سند باید مانع تایید نهایی باشد.

## اصلاحیه مالی

اصلاحیه مالی باید:

- مقدار قبلی و مقدار جدید را نگه دارد.
- دلیل اصلاح را ثبت کند.
- شخص درخواست‌کننده و تاییدکننده را ثبت کند.
- اگر اثر نقدینگی دارد، liquidity model را دوباره محاسبه کند.
- اگر با بانک یا رسید در conflict است، به review برود.

## چیزهایی که فعلاً ساخته نمی‌شود

- موتور واقعی هشدار نقدینگی.
- UI مالی یا dashboard نقدینگی.
- auth، roles یا RLS.
- database، migration یا import واقعی.
- اتصال مستقیم به `audit-app`.
