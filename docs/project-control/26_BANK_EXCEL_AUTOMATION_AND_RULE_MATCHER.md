# Bank Excel Automation and Rule Matcher

آخرین به‌روزرسانی: 2026-06-29

## هدف

این سند طراحی می‌کند که در آینده اکسل گردش حساب روزانه چگونه خوانده، نرمال، تحلیل و با قوانین مدیر match شود. این فقط طراحی مستند است و هیچ parser، import واقعی، database، UI، route یا localStorage ساخته نمی‌شود.

## Daily Bank Excel Import

اکسل گردش بانکی روزانه باید ابتدا به یک لایه staging وارد شود، نه database اصلی. هر فایل باید metadata داشته باشد:

- نام بانک یا حساب.
- تاریخ دریافت فایل.
- بازه تاریخ تراکنش‌ها.
- نسخه قالب فایل.
- شخص واردکننده.
- تعداد ردیف‌ها و تعداد ردیف‌های دارای خطا.
- checksum یا شناسه فایل برای جلوگیری از import دوباره.

## Bank Transaction Normalizer

Normalizer باید قبل از هر تصمیم، این کارها را انجام دهد:

- یکسان‌سازی تاریخ و ساعت.
- تبدیل برداشت/واریز به amount و direction.
- پاک‌سازی فاصله‌ها، کاراکترهای اضافی و متن‌های بانکی تکراری.
- جدا کردن شماره پیگیری، شماره چک، شماره قبض و کلمات کلیدی.
- تشخیص حساب مبدا و مقصد اگر در ستون‌ها یا توضیحات آمده باشد.
- علامت‌گذاری ردیف‌هایی که قالبشان با نسخه شناخته‌شده فرق دارد.

## Transaction Description Matcher

Matcher شرح بانکی را با قانون‌های تعریف‌شده توسط مدیر مقایسه می‌کند. این match نباید به تنهایی برای تصمیم‌های حساس کافی باشد.

روش پیشنهادی:

1. normalize متن توضیحات.
2. جست‌وجوی عبارت‌های دقیق و کلمات کلیدی.
3. بررسی مبلغ دقیق یا بازه مبلغ.
4. بررسی جهت تراکنش: برداشت یا واریز.
5. بررسی حساب مبدا، حساب مقصد و طرف حساب احتمالی.
6. بررسی شماره قرارداد، شماره قسط یا شناسه توافق.
7. بررسی بازه تاریخ قابل قبول.
8. محاسبه سطح اطمینان.
9. پیشنهاد اقدام: ثبت، تایید، اتصال، review یا هشدار.

## قانون‌های قابل تعریف توسط مدیر

| قانون | نمونه کاربرد |
|---|---|
| متن یا عبارت داخل توضیحات | تشخیص قسط، اجاره، خرید، مشتری یا فروشنده |
| مبلغ دقیق یا بازه مبلغ | کنترل مبلغ قسط یا پرداخت ثابت |
| نوع تراکنش | برداشت یا واریز |
| حساب مبدا | تشخیص پرداخت‌کننده |
| حساب مقصد | تشخیص دریافت‌کننده |
| طرف حساب احتمالی | اتصال به مشتری، فروشنده یا کارمند |
| شماره قرارداد یا قسط | تایید installment |
| بازه تاریخ قابل قبول | جلوگیری از تایید قسط خارج از موعد |
| وضعیت اقدام | ثبت رویداد مالی، تایید قسط، اتصال به رویداد موجود، نیازمند بررسی مدیر یا فقط هشدار |

## Rule-based Financial Detection

خروجی matcher باید فقط پیشنهاد بدهد، مگر اینکه اطمینان خیلی بالا باشد. تصمیم‌های پیشنهادی:

- Create new financial event
- Attach to existing event
- Confirm installment
- Needs manager review
- Warning only
- Blocked due to conflict

## Installment Auto Confirmation

تایید خودکار قسط فقط وقتی مجاز است که همه شرط‌های زیر برقرار باشند:

- مبلغ دقیق یا در بازه مجاز قانون باشد.
- تاریخ در بازه قابل قبول قسط باشد.
- جهت تراکنش با قسط سازگار باشد.
- شماره قرارداد، شناسه قسط، طرف حساب یا شرح بانکی match قوی داشته باشد.
- duplicate candidate نباشد.
- رسید یا bank match با conflict مواجه نباشد.
- قانون مدیر صراحتاً اجازه auto confirm داده باشد.

موارد با اطمینان متوسط باید به review بروند، نه تایید خودکار.

## Receipt Attachment Link

Receipt Attachment Link باید بین رسید عکس‌دار، bank transaction و financial event ارتباط قابل ردیابی بسازد. نقش آن:

- پیشنهاد اتصال رسید به تراکنش یا رویداد.
- نمایش دلیل match.
- جلوگیری از اتصال چندباره یک رسید.
- ثبت reviewer، زمان تصمیم و علت رد یا تایید.
- علامت‌گذاری اختلاف مبلغ، تاریخ، طرف حساب یا شماره پیگیری.

## سطح تصمیم‌گیری

| سطح | معنی |
|---|---|
| Auto confirm | فقط برای اطمینان خیلی بالا و قوانین غیرحساس مدیر |
| Needs review | برای اطمینان متوسط یا داده ناقص |
| Blocked | برای تناقض مبلغ، تاریخ، رسید یا طرف حساب |
| Manual only | برای موارد حساس یا خارج از قانون‌های مصوب |

## خطرها

- اشتباه در تشخیص شرح بانکی.
- اتصال تراکنش به شخص اشتباه.
- تایید اشتباه قسط.
- ثبت دوباره یک تراکنش.
- اختلاف رسید و بانک.
- تغییر قالب اکسل بانک.
- توضیحات ناقص یا مبهم.

## Bank Excel Format Change Handling

اگر قالب اکسل تغییر کند:

1. import باید به حالت blocked یا needs review برود.
2. ستون‌های ناشناخته و ستون‌های حذف‌شده گزارش شوند.
3. هیچ داده‌ای مستقیم وارد database اصلی نشود.
4. نسخه قالب جدید فقط پس از approval مرکز فرمان قابل استفاده باشد.
