# Financial Review Queue and Decision Workflow

آخرین به‌روزرسانی: 2026-06-29

## هدف

این سند مشخص می‌کند وقتی اتوماسیون مالی آینده مطمئن نیست، آیتم‌ها کجا نگه‌داری شوند، مدیر یا reviewer چه تصمیم‌هایی بگیرد و نتیجه چگونه قابل audit بماند. این طراحی فقط مستند است و هیچ کد اجرایی، route، UI، database، migration، auth یا localStorage ایجاد نمی‌کند.

## Financial Review Queue چیست؟

`Financial Review Queue` صف کنترل انسانی برای موارد مالی مشکوک، ناقص، حساس یا دارای اختلاف است. این صف باید قبل از هر ثبت نهایی، تایید قسط، اتصال رسید یا اصلاح رویداد مالی، موارد پرریسک را برای تصمیم انسانی نگه دارد.

## چه آیتم‌هایی وارد صف review می‌شوند؟

| آیتم | دلیل ورود به review |
|---|---|
| mismatch بین رسید و بانک | مبلغ، تاریخ، شماره پیگیری، مبدا/مقصد یا طرف حساب با هم نمی‌خوانند. |
| duplicate candidate | تراکنش یا رسید شبیه رکورد موجود است و احتمال ثبت دوباره وجود دارد. |
| weak bank match | match فقط بر اساس بخشی از شواهد مثل مبلغ یا شرح بانکی انجام شده است. |
| تراکنش بدون رسید | پرداخت یا دریافت سند پشتیبان کافی ندارد. |
| قسط با اطمینان متوسط | قسط احتمالی است اما برای auto confirm کافی نیست. |
| پرداخت حساس | مبلغ، طرف حساب، نوع رویداد یا موضوع پرداخت نیازمند تایید مدیر است. |
| اصلاحیه مالی | تغییر مبلغ، تاریخ، طرف حساب، وضعیت پرداخت یا تایید پس از ثبت اولیه رخ داده است. |
| اختلاف مبلغ یا تاریخ | اختلاف با بانک، رسید، قرارداد یا بازه مجاز دیده شده است. |
| تغییر قالب اکسل بانک | normalizer نتوانسته قالب را با اطمینان تشخیص دهد. |
| قانون بانکی جدید یا تغییرکرده | rule هنوز history یا approval کافی ندارد. |

## جریان تصمیم

1. آیتم از validator، bank matcher، receipt flow یا liquidity alert وارد صف می‌شود.
2. سیستم دلیل ورود به review و شواهد match/conflict را نمایش می‌دهد.
3. reviewer یا مدیر تصمیم را انتخاب می‌کند.
4. اگر اصلاح لازم است، correction log ساخته می‌شود.
5. تصمیم نهایی روی رویداد مالی، receipt، bank transaction یا installment اعمال می‌شود.
6. هر تصمیم با زمان، نقش، توضیح و نسخه rule در audit trail ثبت می‌شود.

## تصمیم‌های مدیر

| تصمیم | کاربرد |
|---|---|
| approve | تایید آیتم و اجازه ادامه جریان مالی. |
| reject | رد آیتم به دلیل خطا، سند نامعتبر یا mismatch جدی. |
| request correction | درخواست اصلاح مبلغ، تاریخ، طرف حساب، توضیح یا سند. |
| attach to existing event | اتصال رسید یا تراکنش به رویداد مالی موجود. |
| create new event | ایجاد رویداد مالی جدید از داده بررسی‌شده. |
| confirm installment | تایید قسط پس از بررسی انسانی یا match قوی. |
| mark as duplicate | علامت‌گذاری به عنوان تکراری و جلوگیری از ثبت دوباره. |
| ignore as irrelevant | نادیده‌گرفتن تراکنش یا رسید نامرتبط با جریان مالی مستر جم. |

## خروجی‌های مورد انتظار

| خروجی | توضیح |
|---|---|
| `FinancialReviewItem` | آیتم صف review با نوع ریسک، شواهد، منبع داده و وضعیت. |
| `FinancialReviewDecision` | تصمیم مدیر یا reviewer با نتیجه و دلیل. |
| `FinancialCorrectionLog` | سابقه اصلاح مقدار قبلی، مقدار پیشنهادی، دلیل و تاییدکننده. |
| `FinancialManagerNote` | یادداشت انسانی برای توضیح تصمیم‌های حساس یا مبهم. |

## قوانین امنیتی

- هیچ آیتم دارای conflict نباید خودکار تایید شود.
- duplicate candidate نباید خودکار event جدید بسازد.
- قسط با اطمینان متوسط باید review شود.
- پرداخت حساس بدون تایید مدیر نهایی نشود.
- هر تصمیم باید قابل ردیابی، قابل گزارش و قابل rollback مفهومی باشد.

## چیزهایی که فعلاً ساخته نمی‌شود

- UI صف review.
- backend یا database صف.
- auth، role واقعی یا permission.
- notification واقعی.
- import یا تایید خودکار اجرایی.
