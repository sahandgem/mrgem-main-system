# Human-in-the-loop and Confidence Rules

آخرین به‌روزرسانی: 2026-06-29

## هدف

این سند قانون حضور انسان در حلقه تصمیم را برای اتوماسیون و AI مستر جم تعریف می‌کند. اصل مرکزی این است: سیستم باید تا جای امن کار کند، اما موارد حساس، مشکوک، کم‌اطمینان یا دارای تعارض باید به انسان برسند.

## Human-in-the-loop چیست؟

`Human-in-the-loop` یعنی انسان در نقاط مهم کنترل وارد جریان می‌شود: برای تایید، رد، اصلاح، اتصال داده، تشخیص مورد مشکوک یا تصمیم حساس. این مدل جلوی اتوماسیون خطرناک را می‌گیرد و در عین حال کارهای واضح و کم‌ریسک را سریع‌تر می‌کند.

## سطح اطمینان

| سطح | تعریف | رفتار |
|---|---|---|
| High confidence | داده normalize و validate شده، rule معتبر دارد، conflict ندارد و شواهد قوی است. | auto action allowed with audit trail |
| Medium confidence | شواهد قابل قبول است اما یک یا چند نشانه ناقص یا احتمالی دارد. | needs review |
| Low confidence | داده ناقص، مبهم یا ضعیف است. | review or blocked |
| Conflict | شواهد با هم تناقض دارند؛ مثل اختلاف مبلغ، تاریخ، شخص، سند یا وضعیت. | blocked until manager decision |
| Manual only | موضوع ذاتاً حساس است یا policy اجازه اتوماسیون نمی‌دهد. | never auto-confirm |

## چه چیزهایی می‌توانند auto-confirm شوند؟

فقط مواردی که همه شرط‌های زیر را داشته باشند:

- high confidence باشند.
- rule مصوب و نسخه‌دار داشته باشند.
- conflict یا duplicate candidate نباشند.
- اثر حساس روی پول، database، auth، موجودی یا تعهد قانونی نداشته باشند.
- audit trail کامل شامل دلیل، rule version، داده ورودی و زمان تصمیم داشته باشند.
- rollback یا correction path مفهومی برای آن‌ها طراحی شده باشد.

نمونه‌های بالقوه آینده:

- اتصال receipt کم‌ریسک به event موجود با match دقیق.
- تایید قسط با مبلغ، تاریخ، شماره پیگیری و rule کاملاً سازگار.
- پیشنهاد auto-fix غیرحساس برای فرمت متن یا دسته‌بندی، با ثبت log.
- هشدار خودکار بدون تغییر داده.

## چه چیزهایی همیشه نیاز به review دارند؟

- پرداخت حساس.
- اختلاف رسید و بانک.
- duplicate candidate.
- weak match.
- قسط با اطمینان متوسط.
- فایل اکسل با قالب جدید یا ناشناخته.
- اصلاحیه مالی.
- merge/update کالا.
- حذف، reset، restore یا import حجیم.
- تصمیم‌های مربوط به auth، database، migration یا دسترسی.
- هر موردی که اثر بیرونی یا مالی جدی دارد.

## قوانین تصمیم

| قانون | نتیجه |
|---|---|
| High confidence | auto action allowed with audit trail |
| Medium confidence | needs review |
| Low confidence | review or blocked |
| Conflict | blocked until manager decision |
| Manual only | never auto-confirm |

## مرز مدیر

مدیر باید در این نقاط وارد شود:

- تایید یا رد پرداخت حساس.
- تایید قسطی که exact/strong نیست.
- رفع conflict بین سند، بانک، کالا، موجودی یا برنامه.
- تایید rule جدید یا تغییر rule حساس.
- تایید import یا export مهم.
- تایید خروجی‌هایی که به database یا سیستم خارجی اثر می‌گذارند.

## خطرهای قابل کنترل

- اعتماد بیش از حد به AI.
- تصمیم‌گیری فقط بر اساس شباهت متن.
- تایید خودکار داده ناقص.
- ایجاد یا merge رکورد تکراری.
- پنهان شدن خطای انسانی پشت اتوماسیون.
- نبودن ردپا برای تصمیم‌های حساس.

## اصل نهایی

اتوماسیون باید کار مدیر را سبک کند، نه اینکه کنترل را از مدیر بگیرد.
