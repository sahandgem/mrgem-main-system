# Mahak Architecture Knowledge Extraction

آخرین به‌روزرسانی: 2026-06-29

## موضوع

ثبت و استفاده از نتایج تحلیل معماری نرم‌افزار محک در پروژه هوشمندسازی مستر جم.

## هدف

این تحلیل صرفاً برای استخراج دانش معماری و تجربه طراحی انجام شده است. از این پس محک بخشی از معماری پروژه نیست و فقط به عنوان منبع داده‌های تاریخی و مرجع تحلیل نگه‌داری می‌شود.

پروژه مستر جم می‌تواند از تجربه چندین ساله نرم‌افزار محک در طراحی مدل داده، تست import، مهاجرت داده‌های قدیمی و تحلیل رفتار کسب‌وکار بهره ببرد، بدون اینکه وابستگی معماری یا فنی به محک ایجاد شود.

## Architecture Reference

گزارش تحلیل معماری محک به عنوان یکی از اسناد مرجع معماری نگه‌داری می‌شود. این مرجع فقط برای یادگیری، استخراج الگو و اعتبارسنجی طراحی‌های آینده استفاده می‌شود و نباید به وابستگی فنی، schema، query یا view تبدیل شود.

## الگوهای قابل استفاده

| الگو | کاربرد احتمالی در مستر جم |
|---|---|
| Header / Detail Pattern | طراحی سندهای اصلی و اقلام وابسته مثل فاکتور، تولید، رسید یا انتقال. |
| BaseDocument Pattern | ایجاد مفهوم سند مشترک برای رویدادهای مالی، تولیدی و عملیاتی. |
| BaseDocumentItem Pattern | طراحی اقلام سند بدون وابستگی به جدول یا ستون محک. |
| Master Data Pattern | طراحی موجودیت‌های پایه مثل کالا، طرف حساب، سنگ، گروه و مکان. |
| Feature Based Product Model | طراحی ویژگی‌های کالا، سنگ، وزن، اجرت، barcode و گروه. |
| Audit Trail | ثبت تغییرات، تصمیم‌ها، تاییدها، rollback و import history. |
| Status Based Soft Delete | حذف نرم و وضعیت‌محور برای داده‌های حساس و قابل audit. |
| View Based Reporting | ایده لایه گزارش‌گیری خواندنی بدون وابسته شدن به viewهای محک. |
| Multi Currency Ready Design | آمادگی مفهومی برای چندارزی بودن رویدادهای مالی. |

## موارد ممنوع برای انتقال مستقیم

- ساختار جدول‌های محک.
- نام جدول‌ها.
- نام ستون‌ها.
- وابستگی به SQL schema محک.
- وابستگی به queryهای محک.
- وابستگی به viewهای محک.
- کپی کردن constraint، relation یا naming convention فنی محک.
- طراحی هسته مستر جم بر اساس محدودیت‌های فنی محک.

قانون اصلی: فقط ایده‌ها و الگوهای معماری قابل انتقال هستند، نه ساختار فنی مستقیم.

## استفاده مجاز از داده‌های استخراج‌شده

- طراحی مدل داده.
- اعتبارسنجی ساختار هسته.
- تست import اولیه.
- مهاجرت داده‌های تاریخی.
- تحلیل رفتار کسب‌وکار.
- تولید سناریوهای تست برای product، inventory، finance و production.
- بررسی داده‌های قدیمی فقط از مسیر staging، validation و approved import.

## طراحی هسته مستقل

هسته مستر جم باید مستقل از محک طراحی شود. هیچ موجودیت، API، ساختار جدول، query، view، ستون یا محدودیتی از محک نباید در طراحی نهایی تحمیل شود.

محک از این مرحله به بعد فقط نقش «منبع داده تاریخی» و «مرجع استخراج تجربه طراحی» دارد و هیچ وابستگی عملیاتی به آن در هسته مستر جم ایجاد نمی‌شود.

تمام ماژول‌های جدید باید بر اساس اصول زیر توسعه یابند:

- Event Driven Architecture
- AI Ready Architecture
- Domain Driven Design
- Modular Architecture
- Unified Business Model
- Central Business Event Bus

## ثبت دانش

نتایج این تحلیل به عنوان «دانش استخراج‌شده از سیستم‌های موجود» در مخزن دانش پروژه ثبت می‌شود تا در طراحی ماژول‌های زیر استفاده شود:

- مالی
- انبار
- فروش
- خرید
- تولید
- CRM
- منابع انسانی

## نتیجه مورد انتظار

مستر جم از تجربه طراحی و داده‌های تاریخی محک استفاده می‌کند، اما وابستگی معماری یا فنی به محک ایجاد نمی‌کند. هیچ جدول، ستون، query، view یا schema محک وارد هسته مستر جم نمی‌شود.

## قوانین توقف

- اگر کاری نیازمند merge پروژه `mahak-web-version` باشد، باید متوقف شود.
- اگر کاری نیازمند ورود schema، query، view، جدول یا ستون محک باشد، باید متوقف شود.
- اگر کاری نیازمند تغییر database، migration، route، localStorage، auth یا کد اجرایی باشد، باید فاز مستقل و تایید مرکز کنترل داشته باشد.
